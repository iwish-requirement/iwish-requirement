# 公司内部需求管理系统 – 最终版 PRD

本文件为本项目的唯一权威产品需求文档。后续所有设计与开发实现必须以本 PRD 为准。当 UI 原型与本 PRD 冲突时，以本 PRD 为最终裁决。

---

## 1. 产品背景

公司当前使用表格进行需求管理，存在以下问题：

- 信息分散、难以统一管理。
- 部门需求格式不同，无法统一结构化。
- 无权限管理系统，存在安全风险。
- 无评分系统，影响绩效考核。
- 无统计能力，难以做部门/个人产能的客观分析。

目标：开发新一代需求管理系统，实现更专业、更结构化、更可扩展的内部工具。

---

## 2. 产品目标

- 作为公司内部标准化需求流程系统。
- 支持跨部门需求管理、评分管理、统计分析。
- 动态可配置的部门字段、评分维度、权限角色。
- 在桌面和移动端均可顺畅使用，前端采用移动端优先（mobile-first）布局策略。
- 提供结构化数据沉淀，支持后续 AI 辅助自动分类、自动分析、自动总结（重点为部门月度报告）。

## 3. 用户角色

| 角色                                | 权限范围                              | 示例             |
| --------------------------------- | --------------------------------- | -------------- |
| 超级管理员（Super Admin）                | 系统所有功能、配置                           | CTO / 负责人      |
| 部门管理员（Dept Admin）                 | 本部门需求配置、字段配置、评分维度管理、部分统计       | 各部门 leader     |
| 需求执行者（Developer / Designer / Etc） | 自己部门需求相关操作（查看、处理、反馈、被评分）         | 技术部、UI、投放等 |

| 只读用户（Viewer）                      | 只读查看（根据分配权限查看部门/公司级数据）           | 审计、管理层         |

后续可通过自定义角色和权限模板扩展更多角色。

## 5. 权限系统

### 5.1 权限模型要求

- 权限基于功能点（Feature）定义。
- 角色可以自定义：添加权限、去除权限。
- 用户可绑定多个角色（但一个用户只属于一个主部门）。
- 权限受「功能点 + 部门」控制：
  - 技术部员工仅能访问“技术部需求页面”。
  - 部门管理员能修改本部门字段、评分模板等。
  - 是否可查看跨部门数据由权限点决定（例如有 `demand.view_all` 权限即可查看公司级数据）。

### 5.2 功能点示例

```text
- demand.view_all
- demand.view_department
- demand.view_personal
- demand.create
- demand.edit
- demand.delete
- demand.import
- demand.export
- demand.status.update.self   # 允许创建人/执行人更新与自己相关的需求状态
- demand.status.update.any    # 允许管理员更新任意需求的状态

- department.config

- score.create_template
- score.submit
- score.view_department
- score.view_personal
- score.view_all

- stats.company
- stats.department
- stats.personal
```

需求状态更新必须受上述专门权限点控制：
- 普通成员仅在具备 `demand.status.update.self` 权限，且为该需求的创建人或执行人时，才能修改该需求的状态；
- 具备 `demand.status.update.any` 权限的管理员可以修改任意需求的状态。

未来可扩展更多功能点，如：`audit.view`, `admin.user_manage` 等。

## 6. 需求系统

### 6.1 页面路由模式

- 推荐模式 A：一个路由 + 下拉切换部门。
  - 优点：路由稳定，不随部门配置变化而混乱。
  - 进入页面默认展示当前用户所属部门。
- 模式 B（可选）：每个部门单独一个路由 `/demand/[slug]`。
  - 使用 `departments.slug` 字段配置。
  - 访问仍受权限点和 RLS 控制，不能凭猜 slug 越权。

实际实现：
- 前台以模式 A 为主入口。
- 部门管理员可在后台配置 slug，用于内部直达链接，但不绕过权限控制。

### 6.2 动态字段（重点）

不同部门需求字段不同：

- 技术部：提交人、URL、需求级别、设计稿链接、PR 链接等。
- UI 部门：风格、尺寸、素材链接等。
- 销售部：客户名称、客户级别等。

要求：

- 字段通过管理后台配置：
  - 字段 key（内部唯一标识）
  - 字段名（label）
  - 字段类型：`text` / `textarea` / `select` / `number` / `date` / `boolean` 等。
  - 是否必填。
  - 是否用于筛选。
  - 是否允许导出。
  - 排序（order）。
  - 附加配置：如下拉选项、最值限制、校验规则等（存放在 `config`）。
- 动态字段采用「版本化模板」：
  - 每个部门可有多个字段模板版本（`department_field_templates`），其中一个为当前启用版本。
  - 新创建需求使用当前活跃版本；旧需求记录保留当时的 `field_template_id`，方便历史表单还原。

### 6.3 需求状态流转

默认状态：

```text
待处理 → 进行中 → 已延期 → 已完成 → 不处理
```

说明：
- 「待确认」「已关闭」不作为当前版本的默认状态选项，也不出现在前台状态更新控件中；
- 如业务后续有需要，可通过部门配置增加或调整额外状态，但默认内置状态以「待处理 / 进行中 / 已延期 / 已完成 / 不处理」五个为准。

- 各部门可在配置界面自定义状态列表（存储在 `departments.config` 或额外配置表）。
- 前端根据配置动态渲染状态选择控件。

### 6.4 导入导出

- 导入：
  - 支持 CSV / Excel。
  - 流程：选择部门 → 拉取部门当前字段模板 → 下载导入模板 → 填写后上传。
  - 导入时按字段 key 自动映射，字段类型校验。
  - 数据量预期：在 1–2 千行以内，可同步导入处理。
  - 对于错误：
    - 返回含详细错误信息的文件（例如标注出错行、错误原因）。
    - 阻止导入并提示用户重新修正文件。
- 导出：
  - 按当前筛选条件导出 Excel。
  - 支持选择导出字段（默认导出标记为 `exportable = true` 的字段）。

## 7. 评分系统

### 7.1 评分对象与规则

- 评分主体：需求提交人（Submitter）对执行人进行周期性评分。
- 粒度：
  - 不是针对「单条需求」评分，而是：在一个评分周期内，提交人对其「本周期合作过的每个执行人」各给出一次总体评分。
  - 示例：
    - A 提交人对执行人 A 和 B 各提交了一些需求，则在该周期内，A 只需对执行人 A 和 B 各评分一次。

### 7.2 评分周期

- 周期粒度：按「年月」维度，如 `2025-11`。
- 字符串字段 `period` 形如 `YYYY-MM`。

业务规则补充（与当前实际人工流程对齐）：
1. 服务月与评分月的关系：
   - `period` 表示“服务月”，即需求产生的自然月（例如 `2025-12` 表示 2025 年 12 月的需求）。
   - 某服务月产生的需求，在“下一个自然月”的评分窗口内完成评分（例如 12 月需求在 1 月集中评分）。
2. 评分窗口（评分时间段）：
   - 每个服务月对应一个评分窗口，默认配置为：下一个自然月的 1~3 号。
   - 如遇法定节假日，可将评分窗口顺延至节假日后的第一个工作日，或根据财务考核需要适当提前到当月最后一天；
   - 评分窗口起止日期需要在系统中可配置，以保证能适配不同月份、不同节假日的实际情况。
3. 财务考核节奏约束：
   - 财务会在“评分月”的 5 号之前完成考核统计；
   - 因此每个服务月对应的评分窗口必须在目标统计截止日期前结束（例如：最晚不超过评分月 5 号前的某一配置日期）。

系统实现要求：
- 评分任务中的 `period` 固定代表服务月（需求产生月），而非评分实际发生的自然月；
- 系统应支持按服务月生成评分任务，并通过配置的评分窗口控制任务的“可见/可提交”时间；
- 评分窗口结束后，未完成的任务应被标记为 `missed` 或由管理员进行补评处理，具体策略在评分管理后台中进一步配置。

### 7.3 评分模板（按部门）

- 示例：

```text
技术部：沟通效率、完成质量、时间准确性
UI 部门：美术风格、质量、协作
```

- 配置项：
  - 评分项名称（label）。
  - 所属部门（departmentId）。
  - 分值类型：1–5 或 1–10（存于 `items` 的 `max` 字段）。
  - 是否必填（required）。
- 每个部门可配置多个模板，其中一个为当前启用模板。

### 7.4 评分任务（score_tasks）

- 系统在每个评分周期开始/结束前后，为每个「评分人 – 被评分人 – 部门 – 周期」生成一条评分任务。
- 任务字段：
  - `period`: 如 `2025-11`。
  - `scorer_id`: 提交人（谁来评分）。
  - `target_user_id`: 执行人（被评分人）。
  - `department_id`: 所属部门。
  - `template_id`: 当前部门的评分模板。
  - `status`: `pending` / `completed` / `missed` / `reminded` 等。
  - `last_reminder_at`: 最近一次提醒时间（支持手动补发/自动提醒）。
- 唯一约束：同一周期内，同一 `scorer_id + target_user_id` 仅生成一条任务（忽略具体有多少条需求）。

### 7.5 评分记录（score_records）

- 每个评分任务在用户提交后生成一条评分记录：
  - 与 `score_tasks` 一一对应（通过 `task_id`）。
  - 包含：
    - `scores`: json（`{ [label]: value }`）。
    - `comment`: 文本备注。
    - `created_at`。
- 一旦提交，不允许用户修改；如需变更，由管理员通过后台处理（可选策略）。

### 7.6 漏评与补评

- 对于评分周期内遗漏评分的情况：
  - 通过 `score_tasks.status` 标记为 `pending/missed`。
  - 系统或管理员可以触发补评提醒：
    - 更新 `last_reminder_at`。
    - 不删除原任务，保留补评机会。

### 7.7 评分统计

- 部门评分统计：按部门下所有被评分人的平均分、分布等。
- 个人评分统计：展示被评分人的平均分、各维度得分趋势。
- 公司级评分统计：部门之间评分对比、趋势图（按月/季度）。

## 8. 统计系统

### 8.1 全局统计（公司级）

- 全公司需求数量。
- 完成率。
- 平均耗时（基于 `created_at` 与 `finished_at`）。
- 按月份/季度的趋势图。

### 8.2 部门级统计

- 需求数量。
- 需求类型占比（基于动态字段中某一枚举字段，如需求类型/类别）。
- 完成率。
- 平均耗时。
- 延迟率（基于业务规则，可后续细化）。
- 月度趋势图。

### 8.3 个人统计

- 完成数量。
- 平均处理时间（从 `created_at` 到 `finished_at`）。
- 平均评分（结合评分系统）。

> 注：统计的精度要求偏高，故在 `demands` 中明确存储 `finished_at` 等关键时间戳，避免完全依赖日志推算。

## 9. 高保真 UI 结构（关键页面）

技术栈约定：

- 前端：Next.js App Router + TypeScript。
- UI：Shadcn UI + Tailwind CSS。

### 9.1 登录页

- 邮箱 + 密码登录表单。
- Supabase Auth（SSR + 中间件）作为身份认证方案。

### 9.2 Dashboard（首页）

- 显示：
  - 当前用户部门信息。
  - 今日/近期待处理需求数量。
  - 当前评分周期状态（未开始 / 进行中 / 已结束）。
  - 常用入口：
    - 部门需求
    - 我的需求
    - 评分任务
    - 统计页面

### 9.3 部门需求总览页

- 顶部：部门选择下拉（默认当前部门）+ 新建需求 / 导入 / 导出 按钮。
- 左侧：筛选器（状态、时间范围、提交人、以及本部门配置为 `filterable = true` 的字段）。
- 主区域：
  - 动态列表格，列由当前部门字段模板 + 固定列（标题、状态、创建时间、执行人等）构成。
  - 行 hover 展示操作按钮（编辑、删除、分配执行人等）。
- 支持批量选中并执行操作（如批量改变状态、批量导出）。

### 9.4 需求详情页

- 顶部：需求标题、状态标签、常用操作按钮（编辑、变更状态、更多）。
- 左侧：依据 `field_template_id` 渲染表单字段的只读视图，切换到编辑状态时使用同模板渲染可编辑表单。
- 中部：操作记录（时间轴），包括创建、状态变更、字段修改等。
- 右侧：
  - 评论区：支持子评论楼层结构，实时更新（可后续接入 WebSocket）。
  - 附件区：文件列表 + 上传组件（使用 Supabase Storage）。

### 9.5 部门管理（管理员）

- 动态字段管理：
  - 查看/编辑当前部门字段模板列表（版本）。
  - 管理字段：添加/修改/删除、拖拽排序。
- 路由 slug 配置：配置部门访问 slug。
- 流程状态配置：按部门配置状态列表。（当前版本暂不实现，后续有明确需求时再开启）

### 9.6 评分提交页面

- 显示当前评分周期对当前评分人应评分的执行人列表（来自 `score_tasks`）。
- 每个执行人项展开后：
  - 展示该周期合作情况的概览信息（需求数量等）。
  - 按部门评分模板渲染评分表单（星级/滑杆）。
- 所有必填维度填写完才允许提交。

### 9.7 评分统计页面

- 按部门和周期维度展示：
  - 部门平均分。
  - 各维度得分表。
  - 趋势折线图。

### 9.8 用户管理 / 权限角色管理

- 用户管理：
  - 列表字段：真实姓名、邮箱、部门、角色、状态（`pending` / `active` / `disabled`）。
  - 支持操作：
    - 创建用户（管理员手动创建特殊账号）。
    - 审核/激活用户：将自助注册产生的 `pending` 用户审核为 `active`。
    - 编辑用户：修改真实姓名、部门、角色分配。
    - 停用/启用用户：在 `active` 与 `disabled` 之间切换（停用后禁止登录和访问接口）。
  - 字段约束：
    - 邮箱为只读字段，任何角色均不能在前端修改邮箱。
    - 真实姓名仅管理员可编辑，普通用户仅可查看。
- 角色管理：
  - 左侧角色列表。
  - 右侧权限点树形勾选。
  - 支持为角色分配/移除权限点，并在用户编辑页为用户分配多个角色。

### 9.9 系统设置

- 全局配置项。
- AI 功能开关（是否启用 AI 月度报告）。

## 10. API 设计（主要路由）

### 10.1 Auth

```http
POST /api/auth/register
POST /api/auth/login
POST /api/auth/logout
GET  /api/auth/me
```

- 注册仅允许使用公司邮箱，支持配置多个邮箱域名白名单（如 `company.com`, `subsidiary.com`）。
- 用户使用邮箱和密码自助注册后：
  - 通过 Supabase Auth 创建认证用户；
  - 在业务侧 `users` 表中创建一条记录，状态为 `pending`，仅包含邮箱、auth_id 等基础信息。
- 管理员在「用户管理」中审核：
  - 填写/确认真实姓名、部门并分配角色；
  - 将用户状态从 `pending` 修改为 `active` 后，该用户才具备正常访问权限。
- 邮箱为账号主键字段：
  - 任何角色均不能修改邮箱，仅可通过管理员在 Auth 层做极少量特殊处理（不对普通流程开放）。
- 真实姓名与部门信息：
  - 仅允许管理员在后台编辑；
  - 普通用户不可自行修改，以避免私自更名导致统计与审计混乱。

### 10.2 需求系统

```http
GET    /api/demands
POST   /api/demands
GET    /api/demands/:id
PUT    /api/demands/:id
DELETE /api/demands/:id

POST   /api/demands/import
GET    /api/demands/export
```

### 10.3 部门配置

```http
GET    /api/departments
POST   /api/departments
PUT    /api/departments/:id
DELETE /api/departments/:id

GET    /api/departments/:id/fields
POST   /api/departments/:id/fields
```

### 10.4 评分系统

```http
GET  /api/scores/templates
POST /api/scores/templates

GET  /api/scores/my-tasks
POST /api/scores/submit

GET  /api/scores/stats
```

### 10.5 权限系统

```http
GET  /api/roles
POST /api/roles
PUT  /api/roles/:id
POST /api/roles/assign
```

### 10.6 审计与评论、附件（可扩展）

- 审计日志查询：`GET /api/audit-logs`（带筛选）。
- 评论：`GET/POST /api/demands/:id/comments`。
- 附件：`POST /api/demands/:id/attachments`（实际文件走 Supabase Storage 上传，接口记录 metadata）。

### 10.7 Webhook 集成（可选但推荐）

- 目的：将关键业务事件推送到企业微信机器人、内部系统或其他第三方服务。

```http
GET    /api/webhooks          # 查看/筛选已配置的 webhook 订阅
POST   /api/webhooks          # 创建订阅
PUT    /api/webhooks/:id      # 修改订阅
DELETE /api/webhooks/:id      # 删除订阅
GET    /api/webhooks/events   # 查看事件发送历史、失败原因，并支持重试
```

- 订阅配置（`webhook_subscriptions`）：
  - `event_type`：如 `demand.created`, `demand.status_changed`, `user.approved` 等。
  - `url`：回调地址。
  - `secret`：用于签名请求（HMAC）。
  - `enabled`：是否启用。

- 行为约定：
  - 业务事件发生时写入 `webhook_events` 队列表（包含事件类型与 payload）。
  - 后台任务/Edge Function 异步消费 `webhook_events`，按订阅配置向各 URL 发送 HTTP POST 请求。
  - 每次请求携带签名头（如 `X-Signature`），对方可据此校验请求来源与完整性。
  - 失败请求支持重试与简单退避策略，避免短暂故障导致数据丢失。

- 权限约束：
  - 仅超级管理员可创建/修改/删除 webhook 订阅；
  - 其他管理员可只读查看事件发送历史（视权限点配置）。

### 10.8 企业微信（WeCom）集成

- 目标：将关键业务事件（特别是与需求、评论、评分相关）推送到企业微信，支持群机器人通知和绑定到个人账号的定向提醒。

#### 10.8.1 推送场景

- 新需求创建、状态变更（如从“进行中”到“已完成”）。
- 评论相关：
  - 我创建的需求下有新评论；
  - @ 我的评论或回复。
- 评分相关：
  - 评分周期开始提醒；
  - 有待完成的评分任务；
  - 评分周期即将截止提醒。

上述事件统一通过 10.7 所述 `webhook_events` 分发，企业微信只是其中一个特定的订阅方。

#### 10.8.2 机器人 Webhook 集成

- 通过企业微信群机器人提供的 webhook URL 接收通知。
- 自定义的企业微信 webhook 订阅可使用如下约定：
  - `webhook_subscriptions.provider = 'wecom_bot'`；
  - `url` 字段填入企业微信群机器人提供的 webhook；
  - payload 按企业微信机器人消息格式构造（markdown 或 text），包含关键字段：需求 ID、标题、状态、@ 对象、链接等。

#### 10.8.3 个人账号绑定与定向推送

- 用户模型需要支持与企业微信账号建立绑定关系，以便将部分事件推送到个人：
  - 方案 A：在 `users` 表中增加 `wecom_user_id` 字段，保存企业微信用户的唯一标识；
  - 方案 B：单独一张 `user_wecom_bindings` 表，字段包括：`user_id`、`wecom_user_id`、`corp_id`、`bound_at`、`status` 等。
- 绑定方式可以先从管理员后台手动录入开始，后续再支持扫码绑定 / OAuth。绑定流程不在首版强制实现，但需要在模型和接口上预留：
  - 用户信息接口返回当前绑定状态；
  - 通知发送逻辑在检测到用户有 `wecom_user_id` 时，可追加发送一条个人消息（通过企业微信应用或机器人，具体方案在技术设计中细化）。

- 权限约束：
  - 仅具备相应权限的管理员可以配置企业微信 webhook 和查看企业微信相关发送日志；
  - 普通用户只能查看/管理自己的企业微信绑定状态，不可修改其他人的绑定信息。

## 12. 安全性与审计

### 12.1 安全策略

- 所有接口遵循 Supabase RLS 策略，按用户身份和部门限制访问。
- 使用 Supabase Auth session 进行身份验证，并在服务端中间件中校验权限点。
- 所有关键操作（CRUD、权限变更、导入导出）记录审计日志。

### 12.2 审计日志范围

- 必须记录的操作：
  - 需求 CRUD。
  - 评分模板与评分记录相关操作。
  - 用户和角色、权限变更。
  - 导入/导出行为。
- 日志字段：
  - 操作人账号（user_id）。
  - 操作对象类型与 ID（entity_type, entity_id）。
  - 操作行为（action）。
  - 主要变更字段（changed_fields），记录 key → { before, after }。
  - metadata：IP、User Agent 等。

日志保留策略暂不限制，后续可通过归档任务调整。

## 14. 错误码与异常处理

### 14.1 错误码

| 错误码 | 描述        |
| ---- | --------- |
| 4001 | 缺少必填字段    |
| 4002 | 无权限访问     |
| 4003 | 数据不存在     |
| 5001 | 数据库异常     |
| 5002 | AI 服务异常   |
| 6001 | 导入格式错误    |

### 14.2 通用异常处理

- 权限不足：拦截请求，提示“您没有权限执行此操作”。
- 数据不存在：返回到上一页，提示“该数据已被删除或不存在”。
- AI 服务失败：回退到手动流程，并提示“AI 建议生成失败，请手动填写”。
- 导入格式错误：阻止导入，提示“格式错误，请下载模板后重新导入”，并提供错误文件下载。

## 18. 统计与集成能力补齐

### 18.1 概述与范围

- 在现有需求、评分、统计与集成能力基础上，补齐一套围绕「部门动态字段」的统计中台与可视化看板能力。
- 新增 AI 月度报告（部门/个人）、企业微信绑定与通知订阅，以及 Webhook 对外事件推送，统一基于同一套数据视图。
- 本章节为面向后续实现的增量需求，不与前文已有章节冲突，如有差异以本章节为最新准绳。

### 18.2 部门动态字段统计与看板

1. 概念与目标
   - 「部门动态字段」指各部门在提交需求时按模板配置的字段（如客户、项目、网站、素材类型等）。
   - 目标：让这些字段成为一等公民统计维度，可在部门和个人视角下进行灵活聚合、筛选和对比。

2. 字段级统计配置
   - 在管理端提供「字段统计配置」页面，按部门/模板列出所有动态字段，支持：
     - 是否参与统计（开关）；
     - 聚合方式：计数、去重计数、求和、均值等；
     - 是否可作为分组维度（如按客户、项目分组）；
     - 可见范围：全公司可见 / 仅本部门 / 仅个人。
   - 字段统计配置变更需要记录操作日志（操作者、时间、变更前后差异）。

3. 统计数据与中间层
   - 在数据库中增加面向统计的中间层（宽表或物化视图），统一拆解动态字段：按字段 ID/Key + 值结构存储。
   - 保证字段改名、禁用后的兼容性：
     - 改名保留原 ID/Key；
     - 禁用字段历史数据保留，可配置是否在统计中展示。

4. 部门/个人看板
   - 部门看板：
     - 时间维度：支持自然月、服务月、任意自定义区间；
     - 维度：按客户、项目、网站等动态字段分组，展示需求数量、完成率、逾期率、平均耗时等指标；
     - 视图：趋势图、分布图、部门对比图，可保存为「我的视图」或共享视图。
   - 个人看板：
     - 以个人参与的需求为基础，沿用同一套维度和指标；
     - 支持分析个人在不同客户/项目/网站上的工作结构和质量表现。

### 18.3 AI 月度报告（部门/个人）

1. 报告类型
   - 部门 AI 月度报告：指定部门 + 月度范围，输出结构化的部门运营与协作表现总结。
   - 个人 AI 月度报告：指定个人 + 月度范围，输出个人工作结构、效率与质量分析。
   - 时间粒度：以「月」为主，支持自然月与服务月，需与评分周期保持一致。

2. 数据与生成方式
   - 数据来源：基于前述统计中间层，从需求、评分、部门动态字段和 SLA 指标中聚合出核心指标。
   - 生成方式：
     - 按部门/个人维护可配置的 Prompt 模板（口吻、篇幅、侧重点）；
     - 通过异步任务生成，支持手动触发与定时批量生成；
     - 生成任务需具备幂等与重试能力，失败时记录原因并可手动重试。

3. 展示与导出
   - 列表页：按部门/个人和月份展示报告列表及状态（待生成/生成中/成功/失败）。
   - 详情页：
     - 左侧章节导航（概览、工作量、质量、风险与建议等）；
     - 主区域展示图表概览 + AI 文本报告；
     - 支持对比相邻月份核心指标。
   - 导出能力：
     - 支持导出为 Markdown / PDF，用于邮件或汇报材料；
     - 支持一键发送到企业微信（见 18.4）。

4. 权限
   - 部门报告：部门管理员及以上角色可查看，本部门数据默认不可被其他部门普通成员访问。
   - 个人报告：仅本人、直属上级及拥有特权的管理员可查看。
   - 所有访问与导出需要记录审计日志。

### 18.4 企业微信绑定与通知

1. 账号绑定
   - 在个人设置中提供「绑定企业微信」功能：
     - 通过企业微信 OAuth 流程完成授权，建立系统用户 ID ↔ 企微用户 ID 映射；
     - 支持解绑与重新绑定；
     - 管理端可查看全量绑定状态，按部门/角色筛选与导出。

2. 通知订阅
   - 在「通知设置」页面以事件分组开关的形式配置订阅：
     - 需求事件：创建、指派、状态变更等；
     - SLA 提醒：临近截止与逾期提醒；
     - AI 月度报告：生成完成通知（部门/个人）。
   - 每个事件支持：
     - 开关控制是否发送；
     - 模板预览（标题、摘要字段、跳转链接）。

3. 消息内容与跳转
   - 标准化消息结构：
     - 标题：事件类型 + 关键对象名称（如需求标题、报告名称）；
     - 核心字段：部门、负责人、截止时间、状态、评分周期等；
     - 操作按钮：查看详情 / 打开看板 / 查看报告。
   - 点击消息可跳转至系统 H5/PC 对应页面，需保证登录态与权限校验。

4. 安全与监控
   - 管理后台配置企业微信应用必要参数，仅对超级管理员可见；
   - 推送行为需要有完整的发送日志（包含状态码、耗时、失败原因），支持筛选与重试；
   - 对连续失败、额度限制或签名错误情况触发告警。

### 18.5 Webhook 对外集成

1. 配置管理
   - 后台提供「Webhook 管理」：
     - 新增/编辑/删除订阅；
     - 配置目标 URL、订阅事件类型、签名方式与密钥、重试策略（次数、间隔、超时）。
   - 支持启用/停用开关；仅超级管理员可管理订阅。

2. 事件类型（首批）
   - 需求事件：创建、指派、状态变更等；
   - 统计/报告事件：AI 月度报告生成完成、关键指标阈值触发（如逾期率超标等）；
   - 用户/组织事件（可选）：用户创建与状态变更、部门/角色变更。

3. Payload 规范
   - 统一结构示例：
     - `event_type`：事件类型标识；
     - `timestamp`：事件时间戳；
     - `request_id`：事件请求 ID；
     - `data`：业务数据体（需求基础信息、部门动态字段拆解、相关人员、指标等）；
     - `version`：Payload 版本号。
   - 请求通过 HMAC 等方式签名，签名与时间戳存于 HTTP 头部供对端验签。

4. 推送与重试
   - 事件写入内部事件队列，由 Webhook 服务异步推送；
   - 对失败请求按配置策略自动重试，超过最大次数后标记为失败并记录原因；
   - 提供推送日志列表与手动重试入口，支持按事件类型、订阅 ID、状态筛选。

5. 安全
   - 强制使用 HTTPS URL，敏感配置加密存储；
   - 可选 IP 白名单；
   - 对恶意或异常返回（频繁 4xx/5xx）进行熔断与报警。
